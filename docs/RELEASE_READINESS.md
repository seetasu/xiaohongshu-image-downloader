# 发布就绪检查报告

检查日期：2026-06-25

项目名称：小红书图文素材助手

当前扩展版本：`1.1.8`

已检查发布包：`release/xiaohongshu-material-helper-v1.1.8.zip`

## 总体结论

这是一个 Chrome / Edge 的 Manifest V3 浏览器扩展。项目结构简单，不依赖后端服务，没有外部构建系统，也没有第三方运行时依赖。

不过，目前还不建议直接大范围公开发布。主要问题不是“能不能加载”或“代码能不能启动”，而是产品行为一致性、隐私披露、错误恢复和失败后的用户可理解性。

发布建议：**暂缓公开推广，先修复“应在发布前修复”中的问题。**

当前没有发现会导致扩展完全无法加载或无法打包的硬性阻断问题，但存在足够多的用户体验、隐私说明和文档一致性风险。直接发布会带来较高的支持成本。

## 1. 技术栈

### 运行环境

- 浏览器扩展：Chrome / Edge
- 扩展规范：Manifest V3
- 使用语言：JavaScript、HTML、CSS、JSON、Markdown
- 后台脚本：Chrome 扩展 Service Worker（`background.js`）
- 页面注入：内容脚本（`content.js`）
- 用户界面：扩展弹窗（`popup.html`、`popup.css`、`popup.js`）

### 框架

- 无前端框架
- 无构建框架
- 无包管理器配置
- 无打包工具
- 无测试框架

### 存储方式

- `chrome.storage.local`
  - `background.js` 用它记录默认下载文件夹下的素材包名称计数。
- 弹窗中的 IndexedDB
  - 数据库：`xhs-image-organizer`
  - 对象仓库：`settings`
  - 保存用户选择的文件夹句柄和保存模式。
- 浏览器默认下载目录
  - 通过 `chrome.downloads.download` 保存文件。
- 文件系统访问接口
  - 用户选择自定义文件夹时，通过 `showDirectoryPicker()` 和相关文件句柄写入文件。

### 第三方服务

- 无后端服务
- 无统计分析服务
- 无支付服务
- 无本项目自有远程接口
- 会读取以下站点上的媒体地址：
  - `xiaohongshu.com`
  - `xhscdn.com`

## 2. 安装、启动、构建、测试和发布命令

### 安装

项目没有依赖安装步骤。

手动安装方式：

```bash
# Chrome 扩展管理页
open "chrome://extensions/"

# Edge 扩展管理页
open "edge://extensions/"
```

然后开启开发者模式，选择“加载已解压的扩展程序”，加载 `extension/` 文件夹。

### 启动

项目没有本地开发服务器。直接在 Chrome / Edge 中加载未打包扩展即可。

### 构建

项目没有构建步骤，浏览器直接使用源码文件。

### 校验

本次检查使用过的命令：

```bash
node --check extension/content.js
node --check extension/background.js
node --check extension/popup.js
python3 -m json.tool extension/manifest.json >/dev/null
unzip -l release/xiaohongshu-material-helper-v1.1.8.zip
```

### 测试

当前已有自动化测试命令：

```bash
node --test test/*.test.js
node --test e2e/*.test.js
```

发布前仍需要手动覆盖以下场景：

- 在 Chrome 中加载未打包扩展。
- 在 Edge 中加载未打包扩展。
- 打开普通图片帖子。
- 打开多图轮播帖子。
- 打开包含直接 MP4 地址的视频帖子。
- 测试“保存帖子”。
- 测试“只保存图片”。
- 测试默认下载文件夹。
- 测试自定义保存文件夹。
- 测试已经选择过自定义文件夹后，重新打开弹窗仍能记住路径。
- 测试同名素材包自动编号。
- 测试按回车保存。
- 检查保存后的文件内容、顺序和目录结构。

### 打包 / 发布

在 `extension/` 文件夹内执行打包，确保压缩包根目录直接包含 `manifest.json`：

```bash
zip -r ../release/xiaohongshu-material-helper-v1.1.8.zip \
  manifest.json background.js content.js popup.html popup.css popup.js icons \
  -x '*.DS_Store'
```

当前没有自动发布脚本，也没有 GitHub Actions 工作流。

## 3. 主要用户操作路径

### 路径 A：保存完整帖子素材包到默认下载文件夹

1. 用户打开一个小红书帖子。
2. 用户打开扩展弹窗。
3. 扩展读取标题、正文、图片和可直接下载的 MP4 视频。
4. 弹窗默认选择“保存帖子”。
5. 用户保持保存路径为浏览器默认下载文件夹。
6. 用户点击“保存素材包”或按回车。
7. 后台脚本创建自动编号的素材包文件夹。
8. 扩展保存：
   - `images/`
   - 有视频时保存 `videos/`
   - `正文.txt`
   - `原帖链接.html`
   - `截图.png`

### 路径 B：只保存图片 / 媒体

1. 用户打开弹窗。
2. 用户把“保存内容”切换为“只保存图片”。
3. 该保存模式会被记住。
4. 用户保存已选择的素材。
5. 扩展保存已选择的图片，并且当前实现也会保存已选择的视频。

重要不一致：界面写的是“只保存图片”，但当前实现也会保存视频到 `videos/`。

### 路径 C：保存到自定义文件夹

1. 用户点击“选择”。
2. 浏览器打开文件夹选择器。
3. 用户选择文件夹后，扩展把文件夹句柄保存到 IndexedDB。
4. 扩展通过文件系统访问接口写入文件。
5. 如果同名文件夹已经存在，扩展会创建 `名称 2`、`名称 3` 等编号文件夹。

### 路径 D：已选择文件夹被删除后的恢复

1. 用户在浏览器外删除了之前选择的文件夹。
2. 弹窗可能仍显示旧文件夹名，直到用户重置或重新选择。
3. 用户可以点击“默认”清除失效的文件夹句柄。
4. 用户也可以点击“选择”，扩展会先清除旧句柄，再重新选择文件夹。

## 5. 发布风险

## 阻止发布

没有发现会直接阻止扩展加载或发布包生成的技术问题。

但没有硬性阻断，不代表产品已经适合公开发布。

## 应在发布前修复

### 1. 自定义文件夹保存失败时可能留下半成品素材包

严重程度：高

保存到自定义文件夹时，扩展会逐个写入文件。如果后面的图片或视频下载失败，前面已经写入的文件不会回滚。界面只显示“下载失败”，但磁盘上会留下不完整的素材包。

风险：

- 用户可能误以为完全没有保存，或误以为全部保存成功。
- 不完整素材包可能被误用为完整资料。
- 恢复需要用户手动判断和清理。

建议修复：

- 增加 `保存状态.txt` 之类的完成标记；
- 或先写入临时目录，全部成功后再改名；
- 或在失败时写入 `保存失败.txt`，记录失败文件和链接。

### 2. 默认下载文件夹的自动编号不等同于真实文件系统检查

严重程度：中高

默认下载文件夹模式下，同名素材包编号依赖 `chrome.storage.local` 的计数器，而不是检查真实文件夹是否存在。如果用户删除文件夹、清空下载内容，或清空扩展存储，编号可能跳号或重新开始。

风险：

- 用户可能对编号结果感到困惑。
- “已经保存过一次”的判断不是完全准确的文件系统判断。

建议修复：

- 在文档中说明这个限制；
- 建议需要精确编号的用户使用自定义保存文件夹；
- 或默认下载模式改用时间戳后缀。

### 3. 大视频保存不稳定

严重程度：中高

自定义文件夹模式下，视频通过弹窗里的 `fetch()` 下载。大文件可能因为弹窗生命周期、内存限制、跨域策略或登录态变化而失败。

风险：

- 视频帖子可能保存失败或只保存部分内容。
- 大视频可能导致弹窗卡顿。

建议修复：

- 尽量把下载流程移到后台脚本；
- 默认下载目录优先使用 `chrome.downloads.download`；
- 自定义文件夹保存时明确提示“仅支持可直接下载的 MP4，大视频可能失败”。

### 4. 默认下载模式下弹窗关闭不代表下载完成

严重程度：中

默认下载文件夹模式下，弹窗会在后台开始下载后关闭，但不会等待所有文件实际完成下载。

风险：

- 用户看到弹窗关闭，可能误以为全部保存完成。
- 后续下载失败只会在控制台中出现，不会反馈给用户。

建议修复：

- 监听 `chrome.downloads` 事件；
- 或在后台返回更详细的启动结果和失败清单；
- 或在文档中明确“关闭弹窗表示任务已开始，不代表下载已全部完成”。

### 5. README 对发布用户和开发者的说明还不够清楚

严重程度：中

README 说明了手动安装，但没有清楚区分：

- 普通用户如何安装发布包；
- 开发者如何加载源码；
- 当前没有构建步骤；
- 视频和截图的已知限制。

建议修复：

- 增加“给普通用户的安装方式”和“给开发者的安装方式”；
- 增加“已知限制”章节。

## 可以发布后处理

### 1. 没有自动化测试

严重程度：中

项目目前没有单元测试、集成测试或浏览器自动化测试。当前校验主要是语法检查和人工测试。

建议后续处理：

- 增加基于浏览器自动化的弹窗测试；
- 给 URL 规范化、自动编号、媒体过滤等纯逻辑增加测试。

### 2. 没有可复现的发布脚本

严重程度：中低

发布包通过手动 `zip` 命令生成，容易漏文件或打错版本号。

建议后续处理：

- 增加 `scripts/package.sh`；
- 或增加 `Makefile`；
- 或增加发布检查清单。

### 3. 文件命名仍然比较基础

严重程度：中低

图片和视频目前只按数字编号。简单直观，但没有保存封面标记、原始链接映射或素材说明。

建议后续处理：

- 可选增加 `素材说明.txt`；
- 可选增加 `保存清单.json`；
- 但不要在产品定位还不稳定时重新引入过重的元数据。

### 4. 无障碍体验可以增强

严重程度：低

弹窗已有基础标签，但缩略图按钮的可访问性说明不足，视频缩略图也只是文本标记。

建议后续处理：

- 给缩略图增加类似“第 1 张图片，已选择”的说明；
- 增加键盘选择缩略图的支持。

### 5. 浏览器兼容性矩阵未记录

严重程度：低

扩展使用了文件系统访问接口。Chrome 和 Edge 支持该接口，但不同平台和版本可能表现不同。

建议后续处理：

- 记录已测试浏览器版本；
- 说明自定义文件夹选择依赖文件系统访问接口。

## 6. 数据丢失、错误处理、安全和隐私检查

### 数据丢失 / 半成品保存

- 自定义文件夹模式失败时可能留下半成品素材包。
- 默认下载模式关闭弹窗不代表所有文件完成下载。
- 默认下载文件夹的自动编号依赖本地计数，不是真实文件系统检查。
- 文件夹句柄失效时，已增加“默认”恢复方式，但旧名称仍可能让用户困惑。

### 错误处理

- 多数下载失败只写入控制台，没有明确展示给用户。
- 后台只返回 `started` 数量，没有返回每个文件的成功 / 失败情况。
- 自定义文件夹保存失败时只显示笼统的“下载失败”。

### 安全

- 未发现远程脚本加载。
- 未发现第三方包依赖。
- `原帖链接.html` 中的标题和链接已进行 HTML 转义。
- 主机权限限制在小红书和小红书 CDN，没有申请所有网站权限。

### 隐私

- 未发现统计分析或外部数据上传。
- 扩展会读取页面标题、正文、媒体地址，并截取当前可见页面截图。
- 隐私说明已补充视频处理和本地设置 / 计数器说明。
