# 测试说明

本文档说明如何在发布前运行自动化测试。适合非开发者按步骤检查。

## 测试方案

本项目是一个原生 Chrome / Edge Manifest V3 浏览器扩展，没有后端、数据库服务或构建系统。真正用于浏览器加载和发布的扩展源码在 `extension/` 文件夹。

为了降低维护成本，测试使用 Node.js 自带测试工具，不需要安装第三方测试框架，也不依赖真实小红书页面、付费 API 或外部网络。

端到端测试会启动一个本地 HTTPS 测试页面，模拟小红书帖子，并启动本机 Google Chrome 加载扩展。它只访问本机临时端口，不访问真实小红书。

## 重要测试场景

### 核心业务逻辑

- 保存完整帖子时，是否生成 `正文.txt`、`原帖链接.html`、`截图.png`、`images/` 和 `videos/`。
- “只保存图片”模式是否只生成图片文件。
- 图片是否按 `01`、`02`、`03` 的顺序编号。
- 当前可见封面是否能排到第一张，避免封面被保存成 `02`。
- 同名素材包重复保存时是否自动编号。
- 无效媒体地址是否会被跳过。

### 数据保存、读取、修改和删除

- 保存模式是否能写入本地设置。
- 页面刷新或重新打开弹窗后，保存模式是否能读取回来。
- 保存模式修改后是否覆盖旧值。
- 临时设置删除后是否不再读取到旧值。
- 默认下载目录下的素材包计数是否能递增。

### 用户输入异常

- “存储为”为空时是否回退为 `note`。
- 文件夹名包含 `/ : * ? " < > |` 等异常字符时是否会被清洗。
- 很长的文件夹名是否会被截断，避免超过浏览器下载路径限制。
- 原帖标题和链接里包含特殊字符时，保存的 HTML 是否会转义。

### 网络和第三方接口

- 媒体下载失败时，是否不会中断整批任务。
- 下载失败的文件是否不会计入成功数量。
- 视频只识别可直接下载的 MP4 地址。
- 本产品当前没有 AI 请求、AI 密钥或付费 API，因此“AI 请求成功、失败、超时、返回错误格式”不适用。

### 重复点击和重复数据

- 重复保存同名帖子时，是否生成 `标题 2`，避免覆盖前一次素材包。
- 默认下载目录和自选目录分别需要覆盖编号逻辑。

### 导入、导出、备份和恢复

- 当前产品没有导入、导出、备份、恢复功能，因此自动化测试暂不覆盖。
- 如果后续增加素材包清单、配置导出或备份功能，需要补充对应测试。

### 日期、时区和跨天记录

- 当前只在内容采集结果里记录 `collectedAt`，但保存文件不依赖日期分组。
- 现阶段没有跨天业务规则，因此自动化测试暂不覆盖跨天记录。
- 如果后续按日期归档素材包，需要补充时区和跨天测试。

### 页面刷新、应用重启后的数据保留

- 自动化测试覆盖保存模式的本地读写。
- 浏览器重启后的真实目录权限恢复，需要在 Chrome / Edge 中做手动验收。

### 历史修复回归

- 封面编号错误：当前可见封面必须排在 `01`。
- `截图.png` 中文文件名。
- `正文.txt` 中文文件名。
- `原帖链接.html` 能双击打开原帖。
- “只保存图片”模式不保存视频、正文、链接或截图。
- 同名文件夹自动编号。
- 删除旧自选目录后，重新选择路径不应继续引用旧目录。

## 如何运行

先确认电脑已安装 Node.js。建议使用 Node.js 20 或更高版本。

运行核心逻辑测试：

```bash
node --test test/*.test.js
```

运行端到端测试：

```bash
node --test e2e/*.test.js
```

运行全部测试：

```bash
node --test test/*.test.js e2e/*.test.js
```

如果电脑上有 `npm`，也可以运行：

```bash
npm test
npm run test:e2e
npm run test:all
```

端到端测试需要本机安装 Google Chrome，并允许测试临时启动本地 HTTPS 服务和 Chrome 调试端口。如果当前运行环境不允许监听本地端口，端到端测试会显示跳过；这代表环境受限，不代表产品功能已通过端到端验收。

如果测试通过，会看到类似：

```text
# pass ...
# fail 0
```

只要出现 `fail` 大于 0，就不要发布。需要先修复失败项，再重新运行测试。

## 当前自动化覆盖范围

- `background.js`
  - 下载任务生成
  - 默认下载目录自动编号
  - 保存模式过滤
  - 文件名清洗
  - HTML 转义
  - 单个下载失败后的继续处理

- `popup.js`
  - “存储为”清洗
  - 扩展名识别
  - Data URL 转 Blob
  - IndexedDB 设置读写删
  - 自选目录同名文件夹编号

- `content.js`
  - 图片去重
  - 当前封面排序
  - 图片过滤
  - MP4 视频过滤
  - 内嵌脚本里的转义视频地址识别

- 端到端测试
  - 启动本地模拟小红书帖子页面
  - 加载真实 Chrome 扩展
  - 打开真实扩展弹窗
  - 验证默认下载文件夹、默认全选和素材数量
  - 保存完整素材包并检查目录结构
  - 切换“只保存图片”并检查不会保存正文、链接、截图和视频
  - 验证同名素材包自动编号
  - 验证保存模式重新打开后仍然存在
  - 验证空帖子不会误保存

## 仍未覆盖的高风险路径

- Edge 浏览器里的端到端保存流程。
- 真实小红书线上页面结构变化导致的识别失败。
- 浏览器关闭再打开后，已授权自选文件夹句柄是否仍可正常写入。
- 大视频下载过程中的内存占用、失败恢复和部分文件清理。
- 默认下载目录下真实文件夹已存在，但本地计数器丢失或被清空时的编号冲突。
- 用户连续快速点击保存按钮时，真实浏览器环境下是否会产生多组下载任务。
- 浏览器下载权限被拒绝、下载目录不可写、磁盘空间不足等系统级错误。
- 截图接口在不同窗口、隐私模式或权限受限时的行为。

发布前除了运行自动化测试，还应至少手动验收一次 Chrome 和 Edge 的真实保存流程。
